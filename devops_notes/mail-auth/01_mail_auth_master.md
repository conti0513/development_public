# BIMI導入およびSPF境界値検証  
## QA・デプロイ計画書（mail-auth）

---

## 1. 目的

本ドキュメントは、ビジネスサイドからの  
**BIMI（Brand Indicators for Message Identification）導入要望** に対し、

既存のメール基盤  
（ロストテクノロジー化した構成・外部PF依存を含む）への影響を最小限に抑えつつ、  
**RFC 7208（SPF 10-Lookup 制限）を遵守した安全なデプロイ**を実現するための、

- 試験計画（QA）
- 本番反映手順
- ロールバック方針
- 運用上の注意点

を整理することを目的とする。

※ BIMI自体は SPF Lookup 数を増加させないが、  
DMARC enforcement（`p=quarantine / reject`）を前提とするため、  
**既存SPFレコードの境界値検証を必須要件**とする。

---

## 4. 重点検証項目（エンジニア視点）

### 4.1 SPF 10-Lookup 境界値テスト（RFC 7208）

**「たまたま届く状態」を許容しない。**

#### 単体試験

* `spf_counter.sh` による再帰数の自動計測
* [ ] 現行Lookup数の把握
* [ ] BIMI導入前後での差分確認

#### ネガティブ試験

* あえて Lookup数 = 11 のSPFを作成
* 受信側で `PermError` が発生することを確認

---

### 4.2 ロールバック・プロトコル

**戻せない変更はしない。**

1. **TTL短縮**
   変更24時間前に TTL を 60〜300秒へ変更
2. **State保護**
   Terraform実行前に state バックアップを確認
3. **報告ライン確立**
   社内関係者（例：ゾス）へ
   「即時ロールバック可能」であることを完了条件とする

---

## 5. 運用保守

* **DMARC監視**
  週次でRUAレポートを確認し、BIMI非表示送信元を検知
* **SVG資産管理**
  ロゴ更新時は必ず Tiny PS 変換・検証を実施

## 6. 補足：SPFは「誰が」「どこまで」DNSを見るのか

### 実践：Google Cloud DNS（送信側） → M365（受信側・審判）

「Google Cloud DNS で管理しているドメイン」から
「Microsoft 365（Exchange Online）利用者」へ送信するケースを例に、
**SPF判定時のDNS参照フロー**をAAで可視化する。

```text
[ 送信側：あなたのインフラ ]                    [ 受信側：相手(M365)のインフラ ]
(Google Cloud DNS で管理)                      (Exchange Online 審判)
+------------------------+                    +----------------------------+
|  Cloud DNS (権威)      |                    |  M365 受信ゲートウェイ       |
|  (example.com)         |                    |  (SPF認証モジュール)         |
+-----------+------------+                    +------------+---------------+
            ^                                              |
            |                                  (1) メールが届く
            |                                              |
            |           (2) DNS Lookup #1                  |
            +<---------------------------------------------+
            |  example.com の SPF を取得                   | [累計: 1]
            |                                              |
+-----------+------------+                    +------------+---------------+
|  Google Workspace DNS  |                    |  M365 受信ゲートウェイ       |
|  (_spf.google.com)     |                    |                            |
+-----------+------------+                    +------------+---------------+
            ^                                              |
            |           (3) DNS Lookup #2                  |
            +<---------------------------------------------+
            |  include 先を再帰                           | [累計: 2]
            |                                              |
+-----------+------------+                    +------------+---------------+
|  メルマガPF DNS        |                    |  M365 受信ゲートウェイ       |
|  (spf.pf.com)          |                    |                            |
+-----------+------------+                    +------------+---------------+
            ^                                              |
            |           (4) DNS Lookup #3                  |
            +<---------------------------------------------+
            |  IP判定                                     | [累計: 3]
            |                                              |
            |           (5) 最終判定                      |
            +--------------------------------------------> | 合格 / 失敗
```

---

### 実務的なポイント

* **審判は常に受信側（例：M365）**
* Cloud DNSは「最初に見られる看板」にすぎない
* include先DNSの

  * 応答遅延
  * 障害
  * 再帰深度
    がそのまま不達リスクになる

---

## 7. 設計者としてのまとめ

「GoogleのDNSに書いてある」は
**“審判が最初に見る場所”という意味でしかない。**

SPFの成否は、

* 誰が評価するか
* どこまで再帰するか
* その先のDNSが健全か
* キャッシュとレイテンシがどう効くか

で決まる。

この構造を理解していれば、

> 「M365側のSPF判定では、
> 外部PFのDNSレスポンス遅延や再帰回数がリスクになります」

という **一段上の設計指摘** が可能になる。

---

※ SPF Lookup は「送信側」ではなく、
**受信側（M365 / Gmail 等）が審判として外部DNSを再帰的に参照する** 点を常に意識すること。
