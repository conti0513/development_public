# 基幹業務システムの自社AWS移管・運用近代化計画

## 1. プロジェクトの背景と目的

* **脱ベンダーロックイン** 特定ベンダーのAWS環境・保守体制への依存を解消し、不透明なコストを適正化する。
* **資産の主導権回収** 数千万円規模の投資がなされたシステムを、自社で100%把握・コントロール可能な「自社資産」へ再定義する。
* **運用の近代化** 手動・属人的な運用を、AWSのマネージドサービスによる「自動・可視化された運用」へ転換する。

## 2. PH1：自社環境移管（確実な稼働と権利の回収）

基幹システムとして**「止まらないこと」**を最優先とし、現行環境をベースとした **ハイブリッド移行戦略** を採用する。

### 2.1 AMIベースの確実な環境再現

* **戦略**：クリーン構築の理想を追いつつも、レジストリや特定マイナーバージョンのランタイム等、いわゆる「秘伝の依存関係」を漏らさず再現するため、AMI（マシンイメージ）のアカウント間コピーを使用する。
* **パージ（清掃）**：移管直後に不要なベンダーアカウント、古いログ、一時ファイルを整理し、環境を健全化する。

### 2.2 ネットワークと疎通の完全再現（最重要）

* **IPアドレス依存の解消**：レガシー特有の「IPアドレス直書き」通信を洗い出し、新環境IPへの置換、または Route 53 を用いたドメイン制御により解決する。
* **Security Groupによる外壁制御**：基幹システム本体には極力手を加えず、AWSインフラ層のSecurity Groupで必要最小限のポートのみを許可する**ホワイトリスト運用**を徹底する。

### 2.3 主導権の回収

* **DNS / SSL**：TTLを事前に **300秒** に短縮し、迅速な切り替え・切り戻しを可能にする。
* **認証基盤の刷新**：AWS Systems Manager（SSM）を活用し、移管直後にOS認証情報を自社専用にリセット、ベンダー側の「裏口」を完全に遮断する。

## 3. Windowsサーバーの自動運用設計

設計思想は **「手動作業の排除」** と **「アプリ内スケジュールとの共存」**。

### 3.1 更新管理（Patch & Reboot）

* **SSM Patch Manager**：アプリ内の実行スケジュール（バッチ処理等）と競合しない時間帯をメンテナンスウィンドウとして定義し、パッチ適用から再起動までを**全自動化**する。

### 3.2 監視とセキュリティ

* **Amazon CloudWatch**：外形監視（死活監視）およびOSイベントログを収集。異常検知時は即座にSlack通知。
* **Amazon Inspector**：OSおよびミドルウェアの脆弱性を継続的にスキャンし、IPO準備に耐えうるセキュリティエビデンスを可視化する。

### 3.3 バックアップと可用性（正副構成の最適解）

* **AWS Backup**：毎日1回のAMIフルバックアップと世代管理を自動化。
* **15分復旧（RTO指標）**：高額な常時2台稼働（Active-Standby）は採用せず、最新AMIから15分以内に別AZで復旧可能な構成とすることで、コストと可用性のバランスを最適化する。

### 3.4 ネットワーク・アイデンティティの自動修復

* **起動時プロファイル強制変更**：AMIからの復旧時にWindows Firewallが「Public」プロファイルに誤認され通信が遮断されるリスクを排除するため、OS起動時のスタートアップスクリプト（PowerShell）に、ネットワークカテゴリを強制的に「Private」へ変更する処理を実装する。

## 4. PH2：近代化（モダナイズ）ロードマップ

PH1で安定稼働を確立した後、さらなるコスト削減とポータビリティ向上を目指す。

### 4.1 OS依存のリファクタリング

* 将来的な **Linux（Ubuntu / nginx）化** を見据え、パス区切り文字やフォント依存といったOS固有の要素を段階的に排除する。

### 4.2 IaC化（Infrastructure as Code）

* Terraform 等を用いてサーバー構成をコード化し、環境の構成管理を可視化。AMIの中身が再び「ブラックボックス化」することを防止する。

### 4.3 GitHubによるバージョン管理

* ツール内定義ファイルを XML 等でエクスポートし履歴管理。Java開発の知見を活かした**レビュー前提の近代的運用体制**を確立する。

---

## 5. 移行当日の「切り戻し（Rollback）」基準

IPO前の事業継続性を最優先し、以下の項目が未達の場合は即座に旧環境へ切り戻す「GO/NO-GO判断」を徹底する。

* 特定拠点デバイス（プリンター等）との双方向通信（5003番等）の正常性。
* GCP等、外部プラットフォームとの連携（IP制限解除）の正常性。
* 主要な定時バッチの初動確認。

## エンジニアとしての「こだわり」ポイント

> かつてRedmine移行等の現場で、**「OSは立ち上がるのに、依存関係の微差でデータ連携が死ぬ」** という地獄を経験しました。
> だからこそ今回は、AMIという **「今の正解」** を確実にキャプチャしてベースにしつつ、外側のネットワーク（DNS / Security Group / IP）を徹底的に設計・制御することで、基幹システムの **安全かつ迅速な内製化** を完遂させます。
