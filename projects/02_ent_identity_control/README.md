# Case Study 02: 国策レベル企業におけるID・メール運用と統制対応

## 📌 プロジェクト概要

- **Domain:** 国内大手製造業（国策レベル／数万人規模）
- **Environment:** 厳格なガバナンス・監査前提環境
- **Role:** 運用担当（設計済み基盤の安定運用）
- **Primary Stack:**  
  - Microsoft Entra ID（ID / 認証 / 権限）  
  - Mail / Mail Flow（Exchange 系）  

本案件では、  
**既に設計・承認された統制環境のもとで、IDおよびメール基盤の運用を担当**しました。

自由なツール導入や独自自動化は制限されていましたが、  
結果として **運用事故・監査指摘ゼロ** の状態を継続しています。

---

## 🧩 技術的な運用領域（実務）

### Entra ID（ID運用）

- ユーザーライフサイクル管理  
  - 作成 / 変更 / 無効化 / 削除
- 権限付与・剥奪の厳格な手順運用
- 申請内容と実設定の突合確認
- 操作ログ・監査ログの一貫性維持

👉  
「**誰が・いつ・なぜ付与されたか説明できない権限を作らない**」  
ことを最重要視した運用。

---

### Mail / Mail Flow 運用

- メールアドレス管理（付与・変更・削除）
- 配信ルール・Mail Flow の運用管理
- 誤配信・なりすまし・情報漏洩リスクを前提とした確認手順
- 障害・問い合わせ時のトレース対応

👉  
メールは **IDと同等かそれ以上に“事故ると致命的”** なため、  
常に「**説明可能性と再現性**」を意識した運用を実施。

---

## 🏗 運用構造の整理（後追いでの構造化）

以下は、  
**当時の実運用を後から構造として整理した図**です。

※ 新規に設計したものではなく、  
　「なぜこの運用が壊れなかったのか」を説明するための整理図です。

```mermaid
graph TD
    subgraph Governance [統制・規程]
        Policy[IT規程 / ITGC]
        Policy --> Workflow[標準申請フロー]
    end

    subgraph Operation [日常運用]
        Workflow --> Tool[標準ツール]
        Tool --> Entra[Entra ID]
        Tool --> Mail[Mail / Mail Flow]
    end

    subgraph Evidence [証跡]
        Entra --> Log1[操作ログ]
        Mail --> Log2[メール関連ログ]
        Log1 --> Archive[証跡保管]
        Log2 --> Archive
    end
````

### この構造の意味

* **判断は人が行うが、勝手な操作が入り込まない**
* 運用の結果がそのまま監査証跡になる
* 属人化しない（誰がやっても同じ説明ができる）

---

## 🎯 運用で特に意識していたこと

### 1. 「正しい操作」より「説明できる操作」

* なぜその設定が存在するのか
* なぜその権限が必要なのか
  → **必ず説明できる状態を維持**

### 2. 監査対応を“イベント”にしない

* 監査前に慌てて資料を作らない
* 日常運用の積み重ね＝そのまま監査資料

### 3. 例外対応を増やさない

* 一時対応・特例対応を極力作らない
* 例外は必ず正規フローへ戻す

---

## 💡 この経験が現在にどう活きているか

* クラウドやAIを使う場合でも
  **「壊れない前提」「説明できる前提」から設計を考える**
* 監査・セキュリティ案件でも
  **ドキュメント整備・レビュー対応が苦にならない**
* 自動化を行う際も
  「なぜそれで良いのか」を言語化できる

国策レベル企業での濃密な運用経験は、
現在の **監査フレンドリーなクラウド設計・運用思想の土台** になっています。
