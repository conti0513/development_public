## SASE edge device — SMTP egress control design

I reviewed the firewall capabilities of a SASE edge device.
The focus was on LAN FW and WAN FW, and how to control outbound SMTP traffic.

The goal was to block port 25 from the LAN side and allow only authenticated SMTP (port 587 and 465).
Instead of setting this rule at each endpoint, we use the LAN FW on the SASE device.
This avoids lockout risk caused by endpoint configuration errors.

The device controls traffic at L3-L4 level only.
SMTP authentication is an application-layer job — the device does not handle it.
Encryption between the edge device and the cloud PoP is automatic (DTLS/IPSec).

```text
  └───────────────┘              └─────────┬─────────┘           └──────────────┘

                                  [ DTLS/IPSec Tunnel ]
                                  (Transparent Encryption)
```

## What I learned

- LAN FW on a SASE edge device can block outbound port 25 — infrastructure is the right layer for egress control.
- Allowing only 587/465 is not the same as doing SMTP auth — the device enforces ports, not credentials.
- Controlling at the network layer is safer than endpoint-level policy — fewer failure points, no lockout risk.
- Tunnel encryption is automatic — no extra config needed for the WAN path.
