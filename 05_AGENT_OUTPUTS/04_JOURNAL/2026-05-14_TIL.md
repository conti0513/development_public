## SASE edge device — SMTP egress control design

I reviewed the firewall capabilities of a SASE edge device.
The focus was on LAN FW and WAN FW, and how to control outbound SMTP traffic.

The goal was to block port 25 from the LAN side and allow only authenticated SMTP (port 587 and 465).
Instead of setting this rule at each endpoint, we use the LAN FW on the SASE device.
This avoids lockout risk caused by endpoint configuration errors.

The device controls traffic at L3-L4 level only.
SMTP authentication is an application-layer job — the device does not handle it.
Encryption between the edge device and the cloud PoP is automatic (DTLS/IPSec).

## What I learned

- LAN FW on a SASE edge device can block outbound port 25 — infrastructure is the right layer for egress control.
- Allowing only 587/465 is not the same as doing SMTP auth — the device enforces ports, not credentials.
- Controlling at the network layer is safer than endpoint-level policy — fewer failure points, no lockout risk.
- Tunnel encryption is automatic — no extra config needed for the WAN path.

## LAN FW vs WAN FW — SASE Edge Device Architecture

```
+==============================================================================+
|          SASE Edge Device -- Firewall Architecture (Overview)                |
+==============================================================================+

  [ LAN Side ]              [ SASE Edge ]            [ WAN / SASE Cloud ]
                           +----------+
  +--------------+         |          |               +-------------------+
  | Segment A    | ------> |  LAN FW  | ==============>   Cloud PoP       |
  | 192.168.1.x  |  L3-L4  |          |  DTLS/IPSec   |                   |
  +--------------+         |  * Port  |    Tunnel     |     WAN FW        |
                           |  * IP    |               |                   |
  +--------------+         |  * Proto |               |  * Port / IP      |
  | Segment B    | ------> |          |               |  * App (L7)       |
  | 192.168.2.x  |  L3-L4  |          |               |  * URL Filter     |
  +--------------+         +----------+               +-------------------+
                                                              |
                                                   +----------+----------+
                                                   |                     |
                                            [ Internet ]         [ Other Sites ]


+-------------------------------+------------------+---------------------------+
| Item                          | LAN FW           | WAN FW                    |
+-------------------------------+------------------+---------------------------+
| Where applied                 | SASE Edge Device | SASE Cloud (PoP)          |
+-------------------------------+------------------+---------------------------+
| Traffic scope                 | LAN <-> LAN      | Site <-> Site             |
|                               | LAN -> WAN entry | LAN -> Internet           |
+-------------------------------+------------------+---------------------------+
| Layer                         | L3-L4 only       | L4-L7 (App-aware)         |
+-------------------------------+------------------+---------------------------+
| Sees encrypted traffic?       | NO (before tunnel| YES (after PoP decrypt)   |
+-------------------------------+------------------+---------------------------+
| Typical use case              | Port 25 block    | URL filter / App control  |
|                               | Segment isolation| Inter-site access policy  |
+-------------------------------+------------------+---------------------------+
```

## Key clarification

- LAN FW is implemented on the SASE edge device. It runs before the tunnel.
- WAN FW is implemented in the SASE cloud (PoP). It runs after the tunnel is decrypted.
- Both are managed from the same admin console. Only the enforcement point is different.

## SASE Architecture — Full Concept Summary

```
+==============================================================================+
|                    SASE -- Architecture Overview                             |
+==============================================================================+

  SASE = SD-WAN + Security Functions delivered from the Cloud
  "Network and Security merged into one cloud-native service"

  [ Traditional Model ]
  Branch ---- IPSec VPN ---- HQ / DC ---- Internet
              (tunnel)       (FW here)
  Problem: All traffic goes through HQ. Slow. Single point of failure.

  [ SASE Model ]
  Branch --> Nearest PoP (cloud) --> Internet  :  low latency, no HQ bottleneck


+==============================================================================+
|                    Full Data Flow (Socket to Internet)                       |
+==============================================================================+

  [ LAN ]          [ Socket ]          [ Tunnel ]         [ PoP ]         [ Internet ]
  +----------+     +----------+        +----------+     +----------+
  | Segment A| --> |          |        |          |     |          |
  | Segment B| --> | LAN FW   | -----> | DTLS  or | --> | WAN FW   | --> SaaS
  | Segment C| --> | (L3-L4)  | Encypt | IPSec    |Dcyt | URL Filt | --> Web
  +----------+     +----------+        +----------+     | App Ctrl | --> Sites
                                                        +----------+

  Key points:
  * LAN traffic arrives at Socket as plain text
  * Socket applies LAN FW rules (L3-L4), then encrypts
  * PoP decrypts --> can now inspect content --> applies WAN FW (L4-L7)
  * That is why WAN FW can do App / URL control -- PoP sees plain text


+==============================================================================+
|                    Tunnel Protocol -- DTLS vs IPSec                          |
+==============================================================================+

  VPN (concept) = any technology that creates a secure encrypted tunnel
    |
    +-- IPSec  : UDP 500/4500  | stable, traditional, sometimes blocked by FW
    |
    +-- DTLS   : UDP 443       | looks like HTTPS, NAT-friendly

  Socket tries IPSec first. If blocked --> falls back to DTLS automatically.


+==============================================================================+
|                    Encryption Layers                                         |
+==============================================================================+

  +-------------------------------+--------------------------------------+
  | Layer                         | Scope                                |
  +-------------------------------+--------------------------------------+
  | Tunnel (DTLS / IPSec)         | Socket <-> PoP only                  |
  +-------------------------------+--------------------------------------+
  | App (HTTPS / SMTPS etc.)      | End-to-End (client <-> server)       |
  +-------------------------------+--------------------------------------+
  | Both present (e.g. SMTPS)     | Double encryption                    |
  +-------------------------------+--------------------------------------+

  Note: HTTPS stays encrypted even after PoP decrypts the tunnel.
        To inspect HTTPS content, PoP needs TLS Inspection (separate feature).


+==============================================================================+
|                    Firewall -- Responsibilities                               |
+==============================================================================+

  +-------------------------------+------------------+----------------------+
  | Control target                | LAN FW (Socket)  | WAN FW (PoP)         |
  +-------------------------------+------------------+----------------------+
  | Port block (e.g. TCP:25)      | O                | O                    |
  | LAN segment isolation         | O                | -                    |
  | App / URL control             | - (L3-L4 only)   | O (L7)               |
  | Inter-site access policy      | -                | O                    |
  | TLS inspection                | -                | O (optional)         |
  | SMTP auth enforcement         | - (wrong layer)  | - (App layer job)    |
  +-------------------------------+------------------+----------------------+
  | Enforcement point             | Edge device      | Cloud (PoP)          |
  | Policy management             | Same admin console (centralized)       |
  +-------------------------------+------------------+----------------------+
```
