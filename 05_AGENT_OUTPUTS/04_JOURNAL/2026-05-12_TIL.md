## Session 3 — Network Security: Port Control and FW Strategy

### Topic
Port control and firewall design for on-premises Windows VM infrastructure.

### Port 25: Why Not to Open It Directly

```
Port 25 = plaintext + no authentication
→ Spam relay risk
→ Not acceptable even inside a SASE/ZTNA environment
```

Zero trust principle: trust nothing, encrypt everything. Plaintext is a design anti-pattern.

**Modern approach:** Port 587 + SMTP AUTH (authenticated + encrypted)

### Relay Server Pattern

```
Windows VM → Port 25 (internal only)
    → Relay Server
        → Port 587 + Auth
            → External mail
```

* VMs do not send Port 25 traffic outside
* Authentication handled by the relay server only

### Do Not Control FW at the Endpoint

Setting Windows Firewall directly on a server carries a lockout risk:

```
Wrong FW rule → RDP (3389) blocked → Cannot connect remotely
→ Need Session Manager or serial console to recover
→ Depends on infrastructure team → time lost
```

**Solution:** Control at the network layer (SASE platform), not at the endpoint.

### SASE FW: Two Types

| Type | What it controls |
|---|---|
| WAN FW | Traffic between sites and internet |
| LAN FW | Traffic within a site (east-west, VLAN to VLAN) |

Port 25 outbound block and RDP (3389) → WAN FW scope.

> If not explicitly told to the network vendor, it will not be designed in.

### Key Insight: Explicit Communication Matters

```
"RDP is managed by SASE, not Windows FW"
→ Must be stated clearly to the vendor
→ Silence = not designed = gap in coverage
```

This is a standard infrastructure project risk. Assumptions kill.

### Action Items

* Confirm Port 25 approach: relay server or modern auth (587)
* Confirm LAN FW availability in current SASE contract
* Explicitly communicate: RDP control via SASE, not Windows FW
* Document Session Manager recovery as a fallback procedure
