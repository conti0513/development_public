# 1. 全体アーキテクチャ

企業間連携（EDI / API / ファイル転送）では
**TLS + 証明書**で通信を安全にする。

```
┌──────────────────────────┐
│        企業A (発注元)       │
│                            │
│   Cloud VM / EDI Server    │
│                            │
│   秘密鍵                   │
│   サーバ証明書             │
└─────────────┬──────────────┘
              │
              │ TLS通信
              │
              ▼
┌──────────────────────────┐
│      EDI / 中継基盤        │
│   (EDI Gateway / API)      │
└─────────────┬──────────────┘
              │
              │ TLS通信
              │
              ▼
┌──────────────────────────┐
│        取引先企業           │
│       倉庫 / 物流会社       │
└──────────────────────────┘
```

EDIでは

```
TLS
+ 証明書
```

により

```
通信暗号化
接続先確認
接続元認証
```

を実現する。

# 2. 証明書発行の流れ

証明書は次の手順で発行される。

```
① 鍵ペア生成
② CSR作成
③ 認証局へ申請
④ 証明書発行
⑤ サーバへインストール
```

構造

```
Cloud VM (EDI Server)

┌─────────────────┐
│ 鍵ペア生成        │
│                 │
│ 秘密鍵 (保持)    │
│ 公開鍵          │
└───────┬─────────┘
        │
        │ CSR
        │
        ▼

┌─────────────────┐
│   認証局 (CA)    │
│                 │
│ CSR検証         │
│ 証明書発行      │
└───────┬─────────┘
        │
        │ 証明書
        ▼

EDIサーバへインストール
```

重要

```
秘密鍵は外部に出さない
```

そのため

```
CSRはサーバ側で生成
```

する。

タグ

```
#PKI

#CA

```

# 4. FQDN（Common Name / CN）

証明書には

```
CN = Common Name
```

が含まれる。

これは

```
サーバのFQDN
```

を表す。

例

```
edi.example.com
api.example.com
transfer.example.com
```

TLS通信時

クライアントは

```
アクセスURL
```

と

```
証明書CN
```

を比較する。

```
https://edi.example.com
        │
証明書 CN = edi.example.com
```

一致

```
→ 正常
```

不一致

```
→ TLSエラー
```

タグ

```
#FQDN
#CN
#証明書検証
```

# 6. SC試験での論点

SCでは以下のテーマがよく出題される。

| テーマ   | 内容        |
| ----- | --------- |
| PKI   | 公開鍵基盤     |
| CSR   | 証明書発行手続き  |
| 秘密鍵管理 | 鍵漏洩防止     |
| TLS   | 通信暗号化     |
| 証明書検証 | CN一致      |
| 相互認証  | クライアント証明書 |
